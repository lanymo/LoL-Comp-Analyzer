# 데이터셋 분석 노트

## games.csv 스키마

### 메타 컬럼
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `gameId` | int64 | 매치 고유 ID |
| `creationTime` | int64 | 매치 생성 타임스탬프 (ms) |
| `gameDuration` | int64 | 게임 시간 (초) |
| `seasonId` | int64 | 시즌 ID (모두 9 = 시즌 7) |
| `winner` | int64 | **1 = 팀1 승, 2 = 팀2 승** |

### 팀1 챔피언 픽 (t1)
| 컬럼 패턴 | 설명 |
|---|---|
| `t1_champ{1~5}id` | 챔피언 ID |
| `t1_champ{1~5}_sum1` | 소환사 주문 1 ID |
| `t1_champ{1~5}_sum2` | 소환사 주문 2 ID |

### 팀2 챔피언 픽 (t2)
동일 구조.

### 기타 통계
`t1_towerKills`, `t1_inhibitorKills`, `t1_baronKills`, `t1_dragonKills`, `t1_riftHeraldKills`  
`t1_ban{1~5}` (밴 챔피언 ID)

## 규모 요약
- **총 매치:** 51,490
- **팀1 승:** 26,077 (50.7%) / **팀2 승:** 25,413 (49.3%)
- **유니크 챔피언:** 138종
- **한 매치의 쌍 수:** 시너지 C(5,2)=10 + 매치업 5×5=25 = **35쌍**
- **전체 쌍 연산량:** 51,490 × 35 = 약 **180만 쌍**

## 프로젝트 활용 포인트

### ✅ 이 데이터가 완벽한 이유
- "한 행 = 한 매치" 구조 → 계획서 요구사항 100% 충족
- 전처리(group by) 없이 바로 쌍 분해 가능
- 51,490 매치 × R=1000 부트스트랩 → CPU-bound 확보

### 쌍 분해 예시 (팀1: champ A,B,C,D,E vs 팀2: F,G,H,I,J)
```
시너지 쌍 (팀1): (A,B), (A,C), (A,D), (A,E), (B,C), (B,D), (B,E), (C,D), (C,E), (D,E)
매치업 쌍: (A,F),(A,G),...,(E,J) → 25쌍
```

### 서버 메모리 사용량 추정
- 51,490행 × 61컬럼 × 8바이트 ≈ **약 25MB** → 메모리 상주 문제 없음
