# 프로젝트 계획서 (gRPC + OpenMP) — LoL 챔피언 조합 분석 서비스

**스택:** gRPC + OpenMP (C++) | **대상:** 학부생 단독 | **마감:** 2026-06-28

> **설계 철학:** 2단 파이프라인. **1단계(B, 필수)** 가 단독으로 완성품이 되도록 만들고,
> **2단계(C, 보너스)** 는 떼어내도 무너지지 않는 독립 모듈로 얹는다.
> 과제 문서의 원칙 *"smaller but well-evaluated > ambitious but incomplete"* 를 지키기 위한 구조.
>
> **미들웨어 선택의 핵심:** 본 시스템은 "요청-응답 분석 서비스"다. 클라이언트가 챔피언 조합을
> 질의하면 서버가 그 자리에서 무겁게 계산해 응답한다. 통신(gRPC)이 곧 기능이므로,
> 미들웨어가 명목이 아니라 시스템 성립의 필수 요소다.

---

## 1. Abstract & Problem Definition

**대상 애플리케이션:** 대규모 LoL 랭크 매치 통계를 기반으로, 클라이언트의 질의에
실시간으로 응답하는 **챔피언 조합 분석 서버**. 두 종류의 질의를 처리한다.
- "이 챔피언 조합(예: A·B·C)의 승률과 신뢰구간을 분석해줘" → 쌍 기반 통계 응답.
- "현재 픽 상태에서 다음에 무엇을 고르면 승률이 오르나?" → 후보 챔피언별 기대 승률 병렬 평가.
- (보너스) "이 챔피언과 메타적으로 유사한 챔피언은?" → 유사도 기반 군집 응답.

**계산 병목 (Computational Bottleneck):**
- 한 질의를 처리할 때, 관련 매치 전수에 대해 쌍(pair) 단위 승률을 집계해야 함. 매치당 35쌍(시너지 C(5,2)=10 + 매치업 5×5=25)을 수만 매치에 걸쳐 계산.
- **부트스트랩 리샘플링** R=500~1000회로 승률 신뢰구간을 추정 → 질의당 연산이 매치 수와 무관하게 비례 증가, CPU-bound 강제.
- 추천 질의는 후보 챔피언(~150종)마다 위 계산을 반복 → 연산량 폭증.
- (보너스) 유사도 행렬은 챔피언 수 M(~170)에 대해 O(M²).

**기술 선택 이유:**
- **gRPC:** 클라이언트-서버 간 요청-응답을 구조화된 인터페이스(protobuf)로 정의. 분산 시스템의 표준 통신 패러다임(RPC)을 구현하며, 통신이 시스템 기능의 본질이므로 미들웨어가 필수 요소가 됨. 동시 다중 클라이언트 처리로 worker/concurrency scaling 측정 가능.
- **OpenMP:** 질의당 발생하는 무거운 통계 연산(쌍 집계, 부트스트랩, 후보 평가)을 멀티코어에 분배하여 응답 지연(latency)을 단축.

## 2. System Architecture

```
[데이터]      공개 매치 요약 CSV (새 창에서 데이터 추천 예정)
                         │  서버 기동 시 메모리로 로드 (또는 전처리된 통계 적재)
                         ▼
┌──────────────────────────────────────────────────────────────┐
│ [Client]  gRPC 클라이언트 (조합/추천/유사도 질의 발행)          │
│      │  protobuf 직렬화된 요청                                  │
│ ─────┼──────────── 네트워크 경계 (gRPC channel) ──────────────  │
│      ▼                                                          │
│ [Server]  C++ gRPC 서버 (다중 클라이언트 동시 처리)            │
│      │                                                          │
│ ┌────────────────────────────────────────────────────────┐   │
│ │ 1단계 (B) — 필수: 질의당 쌍 기반 승률 분석              │   │
│ │ #pragma omp parallel for                               │   │
│ │  ├─ 관련 매치에서 시너지/매치업 쌍 분해·집계           │   │
│ │  ├─ thread-local 누적 후 병합                          │   │
│ │  └─ 부트스트랩 리샘플 × R회 → 승률 + 신뢰구간          │   │
│ └────────────────────────────────────────────────────────┘   │
│ ┌────────────────────────────────────────────────────────┐   │
│ │ 2단계 (C) — 보너스 (분리 가능 모듈)                     │   │
│ │ #pragma omp parallel for (쌍 평탄화)                   │   │
│ │  ├─ 챔피언 승률 프로파일 벡터 간 유사도 O(M²)          │   │
│ │  └─ 군집화 → 유사 챔피언 그룹 응답                     │   │
│ └────────────────────────────────────────────────────────┘   │
│      │  protobuf 직렬화된 응답                                 │
│      ▼                                                          │
│ [Client]  결과 수신·출력                                        │
└──────────────────────────────────────────────────────────────┘

[측정]   동시 클라이언트 수별 latency/throughput, 질의당 OpenMP 스레드별 speedup
```

**스레드/프로세스 조직:** Docker Compose로 서버·클라이언트를 독립 컨테이너로 격리(또는 멀티 프로세스/포트로 로컬 시뮬레이션 — 명세서 허용). 서버는 gRPC 워커 스레드로 동시 요청을 받고, 각 요청 내부에서 OpenMP 스레드가 코어 수만큼 생성되어 통계 연산을 병렬화. **즉 두 층위의 병렬성(요청 수준 동시성 + 연산 수준 병렬화)이 공존** — 보고서 분석 포인트.

## 3. Implementation Details

### 3.1 미들웨어 & 직렬화
- **gRPC 서비스 정의(.proto):** `AnalyzeComposition(req) → resp`, `RecommendPick(req) → resp`, (보너스) `FindSimilar(req) → resp`.
- **직렬화:** Protocol Buffers (protobuf). 요청은 챔피언 ID 리스트, 응답은 승률·신뢰구간·랭킹.
- **동시성 설정:** gRPC 서버의 워커 스레드 수, 동시 클라이언트 수를 실험 변수로 변화.

### 3.2 1단계 (B) — 쌍 기반 승률 분석 (질의당)
- 서버는 기동 시 매치 데이터를 메모리에 로드. 질의가 오면 관련 매치에서 시너지 10쌍 + 매치업 25쌍을 해시 키로 집계.
- 부트스트랩: 매치 인덱스를 R회 복원추출하여 승률 신뢰구간 추정. **연산 무게의 1차 원천.**
- 병렬화: `#pragma omp parallel for`로 매치(또는 리샘플) 분배, thread-local 누적 후 병합(atomic 최소화).

### 3.3 2단계 (C) — 유사도 + 군집화 (보너스)
- 챔피언별 승률 프로파일 벡터 구성 → 챔피언 쌍 (i,j), i<j 마다 코사인/유클리드 거리 → M×M 행렬.
- 병렬화: 상삼각 쌍을 1차원 인덱스로 평탄화 후 `#pragma omp parallel for schedule(static)`. 행렬 쓰기 충돌 없어 동기화 불필요.
- **분리 가능성:** 2단계는 1단계 산출물(승률 벡터)만 입력으로 받으므로, 시간이 부족하면 제거해도 1단계 서비스가 독립 완성품으로 남는다.

### 3.4 병렬성 두 층위 비교 (보고서 차별점)
| | 요청 수준 (gRPC 동시성) | 연산 수준 (OpenMP) |
|---|---|---|
| 단위 | 클라이언트 요청 | 매치 / 쌍 / 리샘플 |
| 담당 | gRPC 워커 스레드 | OpenMP 스레드 |
| 측정 | 동시 요청 수 vs latency/throughput | 스레드 수 vs speedup |
| 경합 | 읽기 전용 데이터 공유(경합 적음) | thread-local 후 병합 |

→ 요청 동시성과 연산 병렬화가 한 시스템에 공존 → 자원 경쟁(코어를 두 층위가 나눠 씀) 분석이 풍부해짐.

## 4. Performance Evaluation & Bottleneck Analysis

**필수 실험:**
- **Thread scaling:** 질의 1건을 1/2/4/8 OpenMP 스레드로 처리할 때 speedup.
- **Concurrency scaling:** 동시 클라이언트 1/2/4/8/16일 때 평균 latency·throughput.
- **연산 강도 실험:** 부트스트랩 R=0/100/500/1000으로 늘리며 병렬 이득 증가 곡선.

**필수 그래프:** Speedup vs OpenMP 스레드 수, Latency vs 동시 클라이언트 수, Throughput vs 동시 클라이언트 수.

**병목 분석:**
- protobuf 직렬화·역직렬화 오버헤드 vs 순수 연산 시간 비율.
- 부트스트랩 OFF(가벼움) → ON(무거움) 시 병목이 통신/직렬화에서 CPU로 이동하는 지점 — **보고서 하이라이트.**
- 요청 동시성과 OpenMP 병렬화가 **같은 코어를 경쟁**할 때의 상호 간섭 분석(워커 스레드 ↑ vs OpenMP 스레드 ↑ 트레이드오프).
- gRPC 네트워크 지연 vs 연산 지연 비율.

## 6. 리스크 & 완화
- **gRPC 빌드 연동:** proto 컴파일·C++ 라이브러리 링크에서 초보자가 막히기 쉬움. → **가장 먼저 "hello world" 통신을 성공**시켜 연동의 큰 산을 5월 말까지 넘는다. (이론은 수업에서 학습 완료.)
- **스코프 초과:** 2단계·추천 질의를 욕심내다 1단계 미완성 위험. → **1단계 분석 서비스를 6월 중순까지 독립 완성**, 나머지는 분리 모듈로만 추가.
- **연산 가벼움:** 단순 집계만 하면 speedup 평평. → 부트스트랩 R≥500으로 CPU-bound 확보.
- **두 층위 병렬성 간섭:** gRPC 워커와 OpenMP가 코어를 경쟁 → 오히려 분석 소재로 활용(트레이드오프 측정).

## 7. 데이터셋
- **새 대화창에서 데이터 추천을 받아 확정 예정.**
- 요구 조건: **한 행 = 한 매치** 구조로 챔피언 픽 10개와 승패가 한 레코드에 완결될 것(쌍 분해를 전처리 없이 수행하기 위함). 참가자 단위로 분리된 데이터는 group by 전처리가 추가로 필요.
- 5만 규모면 충분(연산 무게는 데이터 크기가 아니라 부트스트랩 R과 유사도 O(M²)에서 발생). 단, 본 안은 서버 기동 시 데이터를 메모리에 상주시키므로, 메모리에 올라갈 규모인지 확인 필요.

## (참고) 핵심 설계 결정 요약 — 이전 검토 결과
- **미들웨어를 Kafka → gRPC로 전환:** 본 문제가 "스트림 처리"보다 "요청-응답 분석 서비스"에 더 자연스럽고, 진로(시스템 SW/HPC)에서 RPC 통신이 핵심이며, gRPC 이론을 이미 학습함. 통신이 곧 기능이라 미들웨어 명목성 문제 해소.
- **연산 로직(쌍 분해·부트스트랩·유사도)은 미들웨어 변경과 무관하게 그대로 유지** — 이전 B+C 설계 재사용.
